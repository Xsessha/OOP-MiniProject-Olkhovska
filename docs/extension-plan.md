# Extension Plan

## Мета
Покращити надійність системи та підготувати її до використання на рівні продакшену, охоплюючи просунуті теми курсу та усуваючи поточні архітектурні прогалини.

## Стратегія
Розширення пріоритезовані для:
- Закриття обов'язкових прогалин (файлове збереження)
- Посилення архітектури та масштабованості



## Extension Points
- Add file persistence
- Add authentication
- Add pricing system


### 1. Файлове збереження 
Замінити In-Memory репозиторії на:
- Збереження на базі JSON (`System.Text.Json`)
- Завантаження даних під час запуску застосунку
- Збереження даних при кожній зміні стану

**Деталі реалізації:**
- Створення `FileCarRepository`
- Створення `FileRentalRepository`
- Повторне використання інтерфейсів (`ICarRepository`, `IRentalRepository`)

**Критерії готовності (Definition of Done):**
- Дані зберігаються між запусками програми
- Покрито інтеграційними тестами

### 2. Асинхронна підтримка (Async Support)
Впровадити:
- `async/await` у репозиторіях
- Асинхронні методи сервісів

**Деталі реалізації:**
- Інтерфейси на базі `Task`
- Асинхронний ввід/вивід (I/O) для файлів

**Критерії готовності:**
- Відсутність блокуючого I/O
- Асинхронний потік покритий тестами


### 3. Система логування (Logging System)
Додати структуроване логування:
- Абстракція `ILogger`
- Реалізація консольного/файлового логера

**Що логується:**
- Дії користувача
- Помилки
- Бізнес-події

**Критерії готовності:**
- Усі критичні операції логуються
- Помилки можна відстежити

### 4. Шар кешування (Caching Layer)
Впровадити кешування для операцій читання:
- `CachedCarRepository` (Патерн **Decorator**)

**Структури даних:**
- `Dictionary` / `HashSet`

**Критерії готовності:**
- Зменшено кількість повторних звернень до даних
- Реалізовано інвалідацію кешу

### 5. Розширення тестування (Test Expansion)
Розширити стратегію тестування:
- Навантажувальне тестування (Stress tests)
- Тестування продуктивності (Performance tests)
- Тестування файлового збереження

**Інструменти:**
- BenchmarkDotNet або власні таймери

**Критерії готовності:**
- Критичні шляхи протестовані під навантаженням
- Файлові операції верифіковані

### 6. Додаткові патерни проєктування
Впровадити (тільки там, де це виправдано):
- **Decorator** (для кешування/логування)
- **Factory** (для створення репозиторіїв)
- **Strategy** (для логіки ціноутворення)

**Критерії готовності:**
- Патерни вирішують реальні проблеми 

### 7. Ціноутворення та розширення бізнес-логіки
Додати:
- `PricingPolicy` (Політика ціноутворення)
- Динамічний розрахунок ціни
- Ціноутворення на основі тривалості оренди

**Критерії готовності:**
- Бізнес-логіка значно виходить за межі звичайного CRUD


**Критерії готовності:**
- Звіт про покриття коду включено до проєкту


## Архітектурний вплив
**Після розширень система отримає:**
- Сильніше розділення відповідальностей (Separation of concerns)
- Покращену масштабованість
- Реальний шар збереження даних (Persistence layer)
- Структуру, наближену до Production-рівня

## План пріоритетів

| Пріоритет | Завдання |

| **Високий (High)** | Файлове збереження (File persistence) |
| **Високий (High)** | Додаткові доменні класи (Pricing, Payment) |
| **Середній (Medium)** | Асинхронна підтримка (Async support) |
| **Низький (Low)** | Кешування (Caching) |
| **Низький (Low)** | Тестування продуктивності (Performance testing) |


## Очікуваний результат
- Повна відповідність вимогам курсу
- Архітектура, готова до Production
- Демонстрація просунутих навичок ООП та проєктування
- Покращена надійність та простота підтримки

## Підсумкова примітка
Усі розширення:
- ґрунтуються на реальних архітектурних потребах
- відповідають вимогам навчальної програми
- можуть бути інтегровані поступово, не ламаючи існуючу систему

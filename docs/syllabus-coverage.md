#  Syllabus Coverage

## Концепції ООП
- **Класи** (`Car`, `Customer`, `Rental`)
- **Інкапсуляція** (контрольований доступ до стану сутності)
- **Конструктори** для ініціалізації сутностей

 **Як це використовується:**
- Доменні сутності інкапсулюють бізнес-дані та правила
- Зміни стану контролюються виключно через методи

## Абстракція
**Інтерфейси:**
- `ICarRepository`
- `IRentalRepository`
- (опціонально) `IRentalService`

 **Як це використовується:**
- Рівень застосунку залежить від абстракцій, а не від реалізацій
- Інфраструктуру можна замінити без впливу на бізнес-логіку

##  Колекції
- `List<Car>`
- `List<Rental>`

**Як це використовується:**
- Зберігання сутностей в оперативній пам'яті (in-memory)
- База для реалізацій репозиторіїв


## LINQ
- Фільтрація доступних автомобілів
- Пошук оренд та автомобілів

**Приклади використання:**
- `.Where(car => car.IsAvailable)`
- `.FirstOrDefault(...)`

**Мета:**
- Декларативні запити до даних
- Чистіша та легша для підтримки логіка

## Узагальнення (Generics)
- Узагальнені патерни репозиторіїв (якщо реалізовано)
- Абстракції доступу до даних для повторного використання

**Мета:**
- Зменшення дублювання коду
- Забезпечення типобезпеки (type safety)
##  Обробка винятків (Exception Handling)
- Валідація вхідних даних
- Забезпечення виконання бізнес-правил

**Оброблені випадки:**
- Недійсний GUID
- Порожній ввід
- Оренда недоступного авто
- Повернення неорендованого авто

## Принципи SOLID
- **SRP (Принцип єдиної відповідальності):** Обробники, сервіси та репозиторії мають чітко розділені обов'язки
- **DIP (Принцип інверсії залежностей):** Залежності впроваджуються через інтерфейси
- **OCP (Принцип відкритості/закритості):** Репозиторії можна розширювати (наприклад, додати `FileRepository`)

## Патерни проєктування
- **Repository Pattern (Репозиторій):** Абстрагує доступ до даних
- **Service Layer Pattern (Сервісний рівень):** Централізує бізнес-логіку

 **Чому використовується:**
- Розділення відповідальності (Separation of concerns)
- Тестованість
- Розширюваність
##  Тестування
**Модульні тести (Unit tests):**
- Доменна логіка
- `RentalService`

**Інтеграційні тести (Integration tests):**
- Повний цикл оренди/повернення

**Негативні тести:**
- Сценарії помилок
 **Роль:**
- Захист бізнес-логіки
- Слугує критерієм якості (quality gate)

## Архітектура
**Чиста архітектура (Clean Architecture - 4 шари):**
- Domain (Домен)
- Application (Застосунок)
- Infrastructure (Інфраструктура)
- Presentation (Представлення)

 **Ключові властивості:**
- Напрямок залежностей спрямований всередину
- Ізоляція бізнес-логіки
- Замінна інфраструктура

## Рефакторинг
**Виконано під час розробки:**
- Виділення сервісного шару
- Впровадження репозиторіїв
- Відокремлення логіки UI (винесено в обробники/handlers)
- Покращення структури залежностей

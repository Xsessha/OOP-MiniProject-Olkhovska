# Посібник розробника

## Архітектура

Проєкт дотримується принципів чистої архітектури:

Domain → Application → Infrastructure → Console

## Рівні системи

### 1. Домен (Domain)
Містить основні бізнес-сутності:
- Car (Автомобіль)
- Rental (Оренда)
- Customer (Клієнт)
 **Особливості:**
- не має залежностей від інших шарів
- містить бізнес-правила

### 2. Застосунок (Application)
Містить бізнес-логіку:
- `RentalService`
 **Відповідає за:**
- оренду автомобілів
- повернення автомобілів
- бізнес-правила
- валідацію сценаріїв

### 3. Інфраструктура (Infrastructure)
Відповідає за збереження даних:
- `InMemoryCarRepository`
- `InMemoryRentalRepository`
 **Особливості:**
- реалізує інтерфейси Domain
- може бути замінена на File/DB без змін бізнес-логіки

### 4. Представлення (Presentation / Console)
Обробляє взаємодію з користувачем:
- `MenuHandler`
- `RentCarHandler`
- `ReturnCarHandler`
- `AvailableCarsHandler`

## Контракти між шарами

Система використовує інтерфейси для ізоляції шарів:
- `ICarRepository` — доступ до автомобілів
- `IRentalRepository` — доступ до оренд
- `IRentalService` — бізнес-логіка оренди

 **Контракти гарантують:**
- відсутність залежності бізнес-логіки від реалізації збереження
- можливість заміни InMemory на File/DB без змін у Domain
- тестованість системи

## Use Cases (основні сценарії)

Система реалізує мінімум 3 use cases:

### 1. Оренда автомобіля
- користувач вибирає авто
- система перевіряє доступність
- створюється Rental
- авто позначається як зайняте

### 2. Повернення автомобіля
- перевірка активної оренди
- оновлення статусу авто
- завершення оренди

###  3. Перегляд доступних авто
- фільтрація доступних автомобілів
- відображення через UI

## Сценарії помилок (Error Handling)

Система обробляє негативні сценарії:
- оренда вже зайнятого авто
- повернення без активної оренди
- неіснуючий автомобіль
- некоректний ввід користувача

 **Реалізація:**
- перевірки стану перед діями
- обробка винятків (Exceptions)
- захист від некоректного GUID

## Використання колекцій і LINQ

У проєкті використовуються:
- `List<T>` — зберігання сутностей
- `LINQ` — фільтрація даних

 **Приклади:**
- пошук доступних авто
- пошук активних оренд
- вибір конкретного запису

##  Generics

Репозиторії реалізовані через generics:
- `IRepository<T>`
- `InMemoryRepository<T>`

 **Переваги:**
- повторне використання коду
- типобезпечність
- розширюваність

## Стратегія тестування (Quality Gate)

Проєкт вважається валідним якщо:
- всі unit-тести проходять
- інтеграційні сценарії працюють
- негативні сценарії не падають без обробки

## Ітераційна модель розвитку

### Ітерація 1
- базові сутності (Car, Customer)

### Ітерація 2
- `RentalService`
- бізнес-логіка оренди

### Ітерація 3
- Console UI
- меню та взаємодія

### Ітерація 4
- тестування
- рефакторинг
- фінальна документація

 **Кожна ітерація містить:**
- вхідні артефакти
- нову функціональність
- оновлення документації

## Рефакторинг

У процесі розвитку проєкту:
- бізнес-логіку винесено з UI
- додано сервісний шар
- введено репозиторії
- застосовано DIP (через інтерфейси)

## UML-модель

Система базується на 3 основних сутностях:
- `Car`
- `Customer`
- `Rental`

та сервісі:
- `RentalService`

 **Взаємодія:**
- через сервіси
- через репозиторії
- без прямої залежності між шарами

## Висновок

Поточна архітектура забезпечує:
- SOLID принципи
- LINQ + Generics
- Clean Architecture
- тестованість
- розширюваність
